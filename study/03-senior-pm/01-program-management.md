# Program Management vs Project Management

> **Giai đoạn:** Senior PM (5+ năm)
> **Độ quan trọng:** ⭐⭐⭐ Bắt buộc
> **Thời gian học:** 2 tuần lý thuyết + thực hành trên tình huống thực

---

## Sự Khác Biệt Cốt Lõi

Project Manager hỏi: "Làm thế nào để deliver đúng scope, đúng thời hạn, đúng ngân sách?"

Program Manager hỏi: "Deliver để đạt được business outcome gì?"

Đây không chỉ là sự khác biệt về quy mô — đây là sự khác biệt về **mindset** và **horizon**.

---

## Bảng So Sánh Chi Tiết: Project Manager vs Program Manager

| Tiêu chí | Project Manager | Program Manager |
|---|---|---|
| **Phạm vi** | Một dự án cụ thể | Nhiều dự án liên quan, cùng mục tiêu chiến lược |
| **Tầm nhìn** | Deliver trong deadline | Realize benefits sau khi dự án kết thúc |
| **Success metric** | On-time, on-budget, on-scope | Business outcomes đạt được (ROI, market share, cost savings) |
| **KPI chính** | CPI, SPI, scope variance | Benefits realization rate, strategic alignment score |
| **Stakeholder** | Client, project team, sponsor | C-suite, Board, cross-functional leaders |
| **Horizon** | Ngắn–trung hạn (3–18 tháng) | Trung–dài hạn (1–5 năm) |
| **Focus** | Execution và delivery | Governance, strategy, và integration |
| **Team** | Direct management | Manage through Project Managers |
| **Budget** | Project budget | Program budget (tổng hợp nhiều dự án) |
| **Risk** | Project-level risks | Cross-project risks, strategic risks |
| **Change** | Change control trong dự án | Strategic changes ảnh hưởng program direction |
| **Dependencies** | Trong dự án | Giữa các dự án trong program |
| **Conflict** | Team-level | PM-to-PM, cross-functional conflicts |

## Program Là Gì?

Program là nhóm các dự án liên quan được quản lý tích hợp để đạt được **business benefits** mà không thể đạt được nếu quản lý từng dự án riêng lẻ.

**Ví dụ đơn giản:**

```
Mục tiêu chiến lược: "Cải thiện customer experience để tăng retention 20%"

PROGRAM: Customer Experience Transformation

  ├── Project A: CRM Implementation (6 tháng)
  │   Deliverable: Hệ thống CRM tập trung dữ liệu khách hàng
  │
  ├── Project B: Customer Support Chatbot (4 tháng)
  │   Deliverable: AI chatbot giải quyết 60% queries tự động
  │
  ├── Project C: Loyalty Program (3 tháng)
  │   Deliverable: Hệ thống điểm thưởng và reward
  │
  └── Program-level: Change Management & Training
      Deliverable: Nhân viên adopt công cụ mới

BENEFIT: Retention tăng từ 70% → 84% → Strategic goal achieved
```

Nếu quản lý 3 dự án độc lập, chúng sẽ không tích hợp với nhau — CRM không nói chuyện với chatbot, chatbot không biết loyalty points. Program Manager đảm bảo integration xảy ra.

## Program vs Portfolio

```
┌─────────────────────────────────────────────────────────────────┐
│                    ORGANIZATIONAL HIERARCHY                     │
├─────────────────────────────────────────────────────────────────┤
│ PORTFOLIO (Portfolio Manager)                                   │
│ "Đầu tư vào đâu để maximize strategic value?"                  │
│   │                                                             │
│   ├── Program A: Digital Transformation                        │
│   │     ├── Project 1: CRM                                     │
│   │     ├── Project 2: ERP                                     │
│   │     └── Project 3: Mobile App                              │
│   │                                                             │
│   ├── Program B: Market Expansion                               │
│   │     ├── Project 4: Southeast Asia launch                   │
│   │     └── Project 5: B2B channel development                 │
│   │                                                             │
│   └── Standalone Project C: Office Renovation                  │
│         (Không thuộc program nào)                               │
└─────────────────────────────────────────────────────────────────┘
```

| | Program | Portfolio |
|---|---|---|
| **Mục tiêu** | Dự án liên quan, cùng strategic outcome | Tất cả dự án của tổ chức |
| **Quản lý** | Integrated management — dự án phối hợp nhau | Prioritization và resource allocation |
| **Câu hỏi chính** | "Các dự án này có đang hỗ trợ lẫn nhau không?" | "Chúng ta đang đầu tư đúng danh mục dự án không?" |
| **Success** | Benefits realized | Strategic objectives achieved, ROI maximized |

## Program Benefits Realization Management (5 Giai Đoạn)

Benefits Realization là lý do program tồn tại. Nhiều tổ chức deliver projects nhưng không realize benefits — đây là failure của program management.

```
Giai đoạn 1: BENEFITS IDENTIFICATION
  - Xác định benefits từ business case
  - Quantify: "Giảm 30% operational cost" = X triệu/năm
  - Assign benefit owner (người chịu trách nhiệm realize benefit)

         ↓

Giai đoạn 2: BENEFITS PLANNING
  - Benefits Realization Plan: ai làm gì, đo lường thế nào
  - Benefits Map: link từng project deliverable đến benefit
  - Baseline measurement: đo trạng thái hiện tại trước khi program

         ↓

Giai đoạn 3: BENEFITS DELIVERY
  - Ensure projects deliver đúng capability để enable benefits
  - Transition planning: org và process change đi kèm
  - Change management để people adopt new capabilities

         ↓

Giai đoạn 4: BENEFITS TRANSITION
  - Hand over sang Operations
  - Operations "owns" benefits realization sau này
  - Ongoing measurement setup

         ↓

Giai đoạn 5: BENEFITS SUSTAINMENT
  - Đo lường thực tế sau 3, 6, 12 tháng
  - Benefit review với Sponsor
  - Lessons learned: benefit realized hay không? Tại sao?
```

## Program Governance

### Program Board (Steering Committee)

```
PROGRAM BOARD
├── Program Sponsor (Executive Champion) — Quyền cao nhất
│     Chịu trách nhiệm trước Board về business benefits
│
├── Business Change Manager — "Người nhận benefits"
│     Đảm bảo business operation thay đổi để absorb deliverables
│
├── Program Manager (Chair) — Điều hành
│     Báo cáo program progress, escalate issues
│
├── Senior User Representative — Tiếng nói của người dùng
│     Đảm bảo deliverables phù hợp với user needs
│
├── Senior Supplier Representative — Tiếng nói của delivery team
│     Đảm bảo deliverables feasible và high quality
│
└── Finance Representative — Budget governance
      Approve budget release, track spend vs plan
```

### Escalation và Decision Authority

```
Level 1 — Project Manager:
  Tự quyết các vấn đề trong dự án, trong tolerance đã agree
  (±10% budget, ±5 ngày timeline)

Level 2 — Program Manager:
  Quyết định khi vượt project tolerance, cross-project impacts,
  resource conflicts giữa các dự án

Level 3 — Program Board:
  Major scope changes, budget exceptions, strategic direction changes,
  program-level risks

Level 4 — Corporate / Portfolio Board:
  Program-level exceptions, strategic pivots, program termination
```

## Interdependency Management

### Dependency Types

| Type | Mô tả | Ví dụ |
|---|---|---|
| **Mandatory** | Bắt buộc về mặt technical/logic | CRM phải xong trước khi chatbot tích hợp customer data |
| **Discretionary** | Best practice nhưng không bắt buộc | Nên train sales team sau khi có CRM, nhưng có thể trước |
| **External** | Phụ thuộc vào bên ngoài program | Cần vendor deliver hardware trước khi setup server |

### Dependency Types theo Sequence

| Type | Ký hiệu | Mô tả |
|---|---|---|
| Finish-to-Start (FS) | A→B | B chỉ bắt đầu khi A kết thúc (most common) |
| Start-to-Start (SS) | A→→B | B bắt đầu khi A bắt đầu (có thể overlap) |
| Finish-to-Finish (FF) | A──B | B kết thúc khi A kết thúc |
| Start-to-Finish (SF) | A──→B | B kết thúc khi A bắt đầu (rất hiếm) |

## Resource Optimization Across Multiple Projects

Program Manager phải đảm bảo resources không bị conflict giữa các dự án:

**Resource Conflict Scenario:**

```
Tuần 12–14:
  Project A cần: 2 senior BE devs để finalize API
  Project B cần: 2 senior BE devs để integrate với API
  Nhưng tổng senior BE devs chỉ có: 2 người

→ Program Manager must decide: Sequence or find additional resources?
```

**Resource Optimization Techniques:**

1. **Resource Leveling:** Dịch chuyển task để tránh over-allocation (kéo dài timeline)
2. **Resource Smoothing:** Chỉ dịch chuyển task trong float, không thay đổi critical path
3. **Fast-tracking:** Làm song song các work packages không phụ thuộc nhau
4. **Crashing:** Thêm resource vào critical path task (tốn tiền nhưng nhanh hơn)

## Stakeholder Engagement Ở C-Level

C-level stakeholders có concerns và ngôn ngữ khác với mid-level:

```
┌─────────────────────────────────────────────────────────────────┐
│              C-LEVEL STAKEHOLDER COMMUNICATION                  │
├──────────────┬──────────────────────────────────────────────────┤
│ CEO          │ "Chương trình này đóng góp thế nào vào tăng      │
│              │ trưởng? Market position? Competitive advantage?" │
├──────────────┼──────────────────────────────────────────────────┤
│ CFO          │ "ROI là bao nhiêu? Payback period? Cash flow      │
│              │ impact? Phần nào có thể capitalize?"              │
├──────────────┼──────────────────────────────────────────────────┤
│ CTO          │ "Technical debt? Scalability? Security?           │
│              │ Integration với existing architecture?"           │
├──────────────┼──────────────────────────────────────────────────┤
│ COO          │ "Operational disruption? Change management?       │
│              │ Staff impact? Process changes?"                   │
├──────────────┼──────────────────────────────────────────────────┤
│ Board        │ "Strategic risk? Governance? Compliance?          │
│              │ Business continuity?"                             │
└──────────────┴──────────────────────────────────────────────────┘
```

**Golden Rules khi present cho C-level:**
1. Lead with business outcome, not project status
2. Dashboard, không phải bảng data dày đặc
3. Chuẩn bị 3 options, không chỉ 1 recommendation
4. Know your numbers — không được nói "khoảng" hay "ước tính"
5. Anticipate tough questions, đừng bị surprise

## Program Roadmap vs Project Schedule

**Project Schedule:** Gantt chart chi tiết với tasks, resources, dependencies ở level ngày/giờ.

**Program Roadmap:** High-level view theo quý/tháng, focus vào milestones và benefits delivery.

## Ví Dụ Thực Tế: Chuyển Đổi Số Toàn Công Ty SME VN

**Bối cảnh:** Công ty sản xuất đồ gỗ xuất khẩu tại Bình Dương, 300 nhân viên, doanh thu 120 tỷ/năm. CEO muốn số hóa toàn bộ quy trình để scale và chuẩn bị cho chuỗi cung ứng quốc tế.

**Program: Digital Operations Transformation**

```
Program Goal: Giảm operational cost 25% và tăng delivery accuracy lên 95%
              trong 18 tháng

┌─────────────────────────────────────────────────────────────────────┐
│ DỰ ÁN 1: CRM Implementation (4 tháng)                              │
│ Mục tiêu: Centralize customer data, order tracking                  │
│ Budget: 400 triệu VND                                               │
│ Team: 3 devs + 1 BA + PM                                           │
│ Key deliverable: CRM live với 100% orders tracked digitally         │
│ Dependency: None (foundation project)                               │
└─────────────────────────────────────────────────────────────────────┘
          │ API ready + Customer data migrated (Month 4)
          ↓
┌─────────────────────────────────────────────────────────────────────┐
│ DỰ ÁN 2: ERP Integration (6 tháng, bắt đầu month 3)               │
│ Mục tiêu: Kết nối inventory, production, finance với CRM            │
│ Budget: 800 triệu VND                                               │
│ Team: 5 devs + 2 BA + PM                                           │
│ Key deliverable: Real-time inventory + automated PO generation      │
│ Dependency: CRM API (FS), Customer master data (FS)                 │
└─────────────────────────────────────────────────────────────────────┘
          │ ERP live + Inventory data real-time (Month 9)
          ↓
┌─────────────────────────────────────────────────────────────────────┐
│ DỰ ÁN 3: Mobile App (3 tháng, bắt đầu month 7)                    │
│ Mục tiêu: Warehouse staff và sales team dùng mobile                 │
│ Budget: 300 triệu VND                                               │
│ Team: 2 devs + 1 BA + PM                                           │
│ Key deliverable: iOS/Android app với scan, pick, pack               │
│ Dependency: ERP API (FS), CRM customer data (FS)                    │
└─────────────────────────────────────────────────────────────────────┘
          │ All systems live + Staff trained (Month 12)
          ↓
┌─────────────────────────────────────────────────────────────────────┐
│ BENEFITS MEASUREMENT (Month 12–18)                                  │
│ KPI 1: Operational cost reduction → Measure vs baseline             │
│ KPI 2: Delivery accuracy rate → Measure order fulfillment %         │
│ KPI 3: Inventory accuracy → Measure physical vs system count        │
└─────────────────────────────────────────────────────────────────────┘
```

**Key Risks trong Program:**

| Risk | Impact | Probability | Mitigation |
|---|---|---|---|
| CRM chậm trễ → cascade delay cho ERP và Mobile | High | Medium | CRM được prioritize, buffer 2 tuần |
| ERP integration với legacy system phức tạp hơn expected | High | High | POC (Proof of Concept) trong tháng 1 |
| Staff resistance to change (đặc biệt warehouse) | High | High | Change management program song song |
| Vendor lock-in nếu chọn sai ERP | Medium | Low | Evaluate 3 vendors, prefer open API |

## Portfolio Management Cơ Bản

### Project Selection Criteria

Khi tổ chức có nhiều project ideas, Portfolio Manager dùng criteria này để prioritize:

```
PORTFOLIO SCORING MATRIX

Criteria            Weight  Project A  Project B  Project C
───────────────────────────────────────────────────────────
Strategic fit         25%      4          3          5
ROI (3-year)          25%      3          5          2
Risk level            20%      4          3          3
Resource availability 15%      3          4          2
Time to value         15%      2          5          3
───────────────────────────────────────────────────────────
Weighted Score              3.35       3.90       3.05
Ranking                       2          1          3

→ Invest in Project B first, then A, defer C
```
