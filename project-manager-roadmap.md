## Tổng Quan Lộ Trình {#tổng-quan}

```
Giai đoạn 0        Giai đoạn 1         Giai đoạn 2          Giai đoạn 3
[Nền tảng]  →→  [Junior PM / Coord]  →→  [PM chính thức]  →→  [Senior / Program]
 Bất kỳ lúc       0 – 2 năm              2 – 5 năm             5+ năm
```

### Lợi thế của Software Developer khi chuyển sang PM

| Kỹ năng sẵn có | Áp dụng vào PM như thế nào |
|---|---|
| Hiểu SDLC (Software Development Lifecycle) | Lập kế hoạch sprint, ước lượng timeline chính xác hơn |
| Đọc hiểu code & technical debt | Giao tiếp tốt hơn với dev team, không bị "mù" kỹ thuật |
| Quen với Agile/Scrum từ phía dev | Chuyển vai trò sang Scrum Master / Agile PM dễ dàng hơn |
| Hiểu cách hệ thống hoạt động | Risk assessment kỹ thuật sâu hơn PM không có nền dev |
| Kỹ năng logic & problem-solving | Phân tích scope, break down công việc, xử lý dependencies |

## Giai Đoạn 0 — Xây Dựng Nền Tảng {#giai-đoạn-0}

### Mục tiêu
Hiểu PM là gì, vai trò làm gì mỗi ngày, và xác định rõ hướng đi (Technical PM, IT PM, hay Business PM).

### Kiến thức cần nắm

#### 1. PM là gì — và KHÔNG phải là gì
- PM **không phải** là "sếp của developer" — PM không có authority trực tiếp
- PM là người **sở hữu kết quả dự án**, không phải quy trình
- PM khác với Product Manager (PdM): PM quản lý **timeline/scope/budget**, PdM quản lý **product vision/roadmap**
- PM khác với Scrum Master: SM phục vụ team, PM phục vụ **deliverable**

#### 2. Iron Triangle (Tam giác ràng buộc)
```
         Scope
          /\
         /  \
        /    \
       /  PM  \
      /________\
   Time        Cost

→ Thay đổi 1 yếu tố sẽ ảnh hưởng đến 2 yếu tố còn lại
```

**Ví dụ thực tế:**
- Khách hàng muốn thêm tính năng (Scope tăng) → phải tăng Timeline HOẶC tăng Budget, hoặc giảm chất lượng
- Deadline bị rút ngắn (Time giảm) → phải cắt Scope hoặc tăng thêm người (tăng Cost)

#### 3. Các phương pháp quản lý dự án

| Phương pháp | Phù hợp khi nào | Ví dụ thực tế |
|---|---|---|
| **Waterfall** | Requirement rõ ràng, ít thay đổi, compliance cao | Xây dựng cầu đường, hệ thống ngân hàng core |
| **Agile / Scrum** | Requirement thay đổi nhiều, cần phản hồi nhanh | SaaS product, mobile app startup |
| **Kanban** | Công việc liên tục, không có sprint cố định | Support ticket, bug fixing team |
| **Hybrid** | Dự án có phần rõ ràng + phần không chắc chắn | Migration project (Delphi → modern stack) |
| **Prince2** | Phổ biến ở UK/EU, formal governance | Dự án chính phủ, tài chính |

### Bài tập giai đoạn 0

**Bài tập 1: Tự phân tích Iron Triangle**
Nghĩ về một dự án bạn từng tham gia với tư cách developer:
- Scope ban đầu là gì? Có thay đổi không?
- Timeline có bị trễ không? Vì lý do gì?
- Budget có vượt không? Ai chịu trách nhiệm?
- Nếu bạn là PM, bạn xử lý khác không? Xử lý thế nào?

**Bài tập 2: Phỏng vấn 1 PM**
Tìm 1 người đang làm PM (LinkedIn, công ty hiện tại) và hỏi:
1. Một ngày làm việc của bạn trông như thế nào?
2. Điều khó nhất trong công việc là gì?
3. Bạn dùng công cụ gì hàng ngày?
4. Kỹ năng nào quan trọng nhất khi mới bắt đầu?

## Giai Đoạn 1 — Junior PM (0–2 năm) {#giai-đoạn-1}

### Vai trò phù hợp để bắt đầu
- **Project Coordinator** — hỗ trợ PM chính, quản lý tài liệu, tracking
- **PMO Analyst** — phân tích dữ liệu dự án, báo cáo, chuẩn hóa quy trình
- **Scrum Master** — dễ tiếp cận với background developer
- **Technical PM / IT PM Junior** — quản lý các dự án IT nhỏ

### Kỹ năng cần phát triển

#### 1. Quản lý thời gian & ưu tiên công việc

**Kỹ thuật MoSCoW Prioritization:**
```
M — Must have    (bắt buộc, không có là fail)
S — Should have  (quan trọng, nên có nhưng có thể delay)
C — Could have   (nice to have, nếu có thời gian)
W — Won't have   (loại bỏ khỏi scope lần này)
```

**Ví dụ áp dụng cho dự án migration:**
- **Must:** Migrate core modules trước deadline Q4
- **Should:** Migrate reporting module trong Q4
- **Could:** Thêm dashboard mới sau khi migrate
- **Won't:** Redesign UI/UX trong giai đoạn này

#### 2. Viết Project Charter (Điều lệ dự án)

Template tối thiểu:
```markdown
## Project Charter: [Tên dự án]

**Project Sponsor:** [Ai tài trợ/phê duyệt]
**Project Manager:** [Tên PM]
**Start Date:** | **End Date:**

### Mục tiêu dự án
[1-2 câu mô tả kết quả mong muốn]

### Scope IN (nằm trong phạm vi)
- ...

### Scope OUT (ngoài phạm vi)
- ...

### Các milestone chính
| Milestone | Deadline | Người chịu trách nhiệm |
|---|---|---|
| | | |

### Budget ước tính
[Con số hoặc range]

### Rủi ro chính
| Rủi ro | Xác suất | Impact | Mitigation |
|---|---|---|---|
| | | | |

### Stakeholders
| Tên | Vai trò | Mức độ ảnh hưởng |
|---|---|---|
| | | |
```

#### 3. Work Breakdown Structure (WBS)

**Cách tạo WBS:**
1. Bắt đầu với deliverable lớn nhất
2. Phân nhỏ thành các work package (mỗi package ≤ 8 giờ công)
3. Assign người chịu trách nhiệm
4. Estimate effort (giờ/ngày)

**Ví dụ WBS cho dự án migration hệ thống:**
```
Migration Project
├── 1. Analysis Phase
│   ├── 1.1 Document current system (3 days)
│   ├── 1.2 Identify dependencies (2 days)
│   └── 1.3 Gap analysis report (2 days)
├── 2. Development Phase
│   ├── 2.1 Module A migration (10 days)
│   ├── 2.2 Module B migration (8 days)
│   └── 2.3 Integration testing (5 days)
├── 3. UAT Phase
│   ├── 3.1 Test case preparation (3 days)
│   ├── 3.2 User acceptance testing (5 days)
│   └── 3.3 Bug fixing (3 days)
└── 4. Go-live
    ├── 4.1 Data migration (2 days)
    ├── 4.2 Cutover (1 day)
    └── 4.3 Hypercare (5 days)
```

#### 4. Giao tiếp với stakeholders

**Stakeholder Communication Matrix:**
| Stakeholder | Quan tâm gì | Báo cáo gì | Tần suất | Kênh |
|---|---|---|---|---|
| Sponsor | Budget, timeline, ROI | Executive summary | Bi-weekly | Email + meeting |
| Dev team | Technical requirement, scope | Sprint planning, blockers | Daily | Slack + standup |
| End user | Tính năng, UX | Demo, training | Per milestone | Meeting |
| Khách hàng | Deadline, cost | Progress report | Weekly | Email |

### Case Study 1: Dự án trễ deadline

**Tình huống:**
Bạn là Project Coordinator cho dự án phát triển phần mềm quản lý kho. Timeline 4 tháng. Sau 2 tháng, dev team báo cáo sẽ cần thêm 6 tuần vì scope thay đổi.

**Thông tin:**
- Khách hàng không đồng ý tăng timeline
- Budget đã cố định
- Dev team đang ở 110% capacity

**Câu hỏi phân tích:**
1. Bạn sẽ làm gì trước tiên?
2. Bạn present vấn đề này với sponsor như thế nào?
3. Các option nào khả thi?
4. Bạn chọn option nào và tại sao?

**Hướng giải quyết tham khảo:**
- Thu thập dữ liệu: scope nào đã thêm vào, ai approve?
- Phân tích: có thể cắt scope nào mà không ảnh hưởng MVP?
- Present 3 options cho sponsor: (A) cắt scope, (B) tăng timeline, (C) tăng resource
- Document mọi quyết định và change request

## Giai Đoạn 2 — PM Chính Thức (2–5 năm) {#giai-đoạn-2}

### Vai trò
- **Project Manager** (IT, Software, Infrastructure)
- **Technical Project Manager**
- **Delivery Manager**
- **Agile Project Manager**

### Kỹ năng nâng cao

#### 1. Risk Management (Quản lý rủi ro)

**Risk Register template:**
| ID | Mô tả rủi ro | Xác suất (1-5) | Impact (1-5) | Score | Chiến lược | Hành động | Owner |
|---|---|---|---|---|---|---|---|
| R001 | Key developer nghỉ việc | 3 | 4 | 12 | Mitigate | Cross-train backup | PM |
| R002 | Third-party API không ổn định | 2 | 5 | 10 | Accept + Contingency | Chuẩn bị fallback | Tech Lead |

**Risk scoring matrix:**
```
Impact →    1       2       3       4       5
Prob ↓
  5        5      10      15      20      25
  4        4       8      12      16      20
  3        3       6       9      12      15
  2        2       4       6       8      10
  1        1       2       3       4       5

Score ≥ 15: High Risk → Escalate ngay
Score 8-14: Medium Risk → Monitor chặt chẽ
Score ≤ 7:  Low Risk → Accept và log
```

#### 2. Earned Value Management (EVM)

EVM là cách đo lường sức khỏe dự án bằng con số, không phải cảm tính.

**Các chỉ số cốt lõi:**
```
PV  = Planned Value     (Giá trị kế hoạch: bạn đã lên kế hoạch làm gì đến hôm nay?)
EV  = Earned Value      (Giá trị thực hiện: bạn thực sự hoàn thành bao nhiêu?)
AC  = Actual Cost       (Chi phí thực tế bỏ ra)

CV  = EV - AC           (Cost Variance:     > 0 = under budget ✅)
SV  = EV - PV           (Schedule Variance: > 0 = ahead of schedule ✅)
CPI = EV / AC           (Cost Performance Index:     > 1 tốt ✅)
SPI = EV / PV           (Schedule Performance Index: > 1 tốt ✅)
```

**Ví dụ thực tế:**
```
Dự án 100 triệu VND, 10 tuần.
Tuần 5: 
  PV = 50 triệu (đáng lẽ xong 50% công việc)
  EV = 40 triệu (chỉ xong 40% công việc)
  AC = 55 triệu (đã tốn 55 triệu)

CV = 40 - 55 = -15 triệu  → Over budget ❌
SV = 40 - 50 = -10 triệu  → Behind schedule ❌
CPI = 40/55 = 0.73         → Mỗi đồng bỏ ra chỉ thu về 0.73 đồng giá trị ❌
SPI = 40/50 = 0.80         → Tốc độ chỉ bằng 80% kế hoạch ❌

EAC (Estimate at Completion) = BAC / CPI = 100 / 0.73 ≈ 137 triệu
→ Dự án sẽ tốn khoảng 137 triệu thay vì 100 triệu!
```

#### 3. Quản lý Change Request

**Quy trình Change Request:**
```
1. Stakeholder yêu cầu thay đổi
         ↓
2. PM ghi nhận vào Change Log
         ↓
3. Phân tích impact: Time? Cost? Scope? Quality?
         ↓
4. Present options cho sponsor
         ↓
5. Sponsor approve / reject
         ↓
6. Nếu approve: update Project Plan, communicate to team
7. Nếu reject: document lý do, notify stakeholder
```

**Change Request Form template:**
```
CR-[số thứ tự]: [Tên thay đổi]
Ngày yêu cầu: | Người yêu cầu:
Mô tả thay đổi:
Lý do:
Impact đến Scope:
Impact đến Timeline: +/- [số ngày]
Impact đến Budget:   +/- [số tiền]
Quyết định: ☐ Approved  ☐ Rejected  ☐ Deferred
Ký bởi (Sponsor):
```

### Case Study 2: Khách hàng khó tính

**Tình huống:**
Bạn đang PM cho dự án xây dựng hệ thống ERP cho công ty logistics. Khách hàng liên tục thêm yêu cầu mới mà không ký change request. Khi bạn hỏi, họ nói "đây là requirement ban đầu, tôi không đồng ý trả thêm".

**Thông tin:**
- Contract ký dựa trên scope document 20 trang
- Khách hàng claim requirement mới nằm trong scope đó
- Dev team đang overload

**Phân tích:**
1. Kéo scope document ra và đối chiếu từng điểm
2. Nếu thực sự unclear → học bài, lần sau viết scope rõ hơn
3. Nếu rõ ràng là ngoài scope → escalate lên account manager/legal

**Bài học:**
- "Scope creep" là rủi ro lớn nhất trong PM
- Mọi yêu cầu thay đổi đều phải đi qua Change Request process
- PM cần biết "nói không" một cách chuyên nghiệp: không phải "không làm" mà là "làm với điều kiện..."

### Case Study 3: Team conflict

**Tình huống:**
Frontend dev và Backend dev trong team của bạn liên tục xung đột về API design. Mỗi người đều cho rằng phía kia cần thay đổi. Dự án bị block 3 ngày.

**Hướng xử lý của PM:**
1. Tổ chức meeting riêng với từng bên để hiểu rõ vấn đề
2. Tổ chức meeting chung, focus vào deliverable (không phải ego)
3. Nếu không giải quyết được → escalate lên Tech Lead hoặc Architecture decision
4. Document quyết định cuối cùng

**Nguyên tắc:** PM không cần biết đúng/sai về kỹ thuật — PM cần đảm bảo team **ra quyết định** và **tiến về phía trước**.

## Giai Đoạn 3 — Senior PM (5+ năm) {#giai-đoạn-3}

### Vai trò
- **Senior Project Manager**
- **Program Manager** (quản lý nhiều dự án liên quan)
- **Portfolio Manager** (quản lý danh mục dự án toàn công ty)
- **PMO Director / Head of PMO**
- **Delivery Director**

### Kỹ năng ở cấp này

#### 1. Program Management vs Project Management

| | Project Manager | Program Manager |
|---|---|---|
| Phạm vi | 1 dự án | Nhiều dự án liên quan |
| Tầm nhìn | Tactical (làm gì, khi nào) | Strategic (tại sao, lợi ích gì) |
| KPI | Deliver đúng scope/time/cost | Deliver business outcome |
| Stakeholder | Project team + direct client | C-level, Board |
| Horizon | Có ngày kết thúc | Liên tục (program ongoing) |

#### 2. OKRs & Strategic Alignment

Ở cấp Senior, PM không chỉ deliver dự án — mà phải đảm bảo dự án **đóng góp vào mục tiêu công ty**.

**Framework OKRs:**
```
Objective: Tăng market share ở mảng logistics phần mềm tại VN
  KR1: Onboard 5 khách hàng mới trong Q3
  KR2: Giảm churn rate từ 15% xuống 8%
  KR3: NPS score ≥ 40

→ Dự án "phát triển tính năng real-time tracking" liên kết với KR2 và KR3
→ PM cần biết đây là lý do dự án tồn tại, không chỉ deliver feature
```

#### 3. Building PMO (Project Management Office)

Khi lên Director/Head level, nhiệm vụ là xây dựng hệ thống PM cho cả tổ chức:
- Chuẩn hóa quy trình và template
- Training PM junior
- Governance: project selection, prioritization
- Reporting lên Board/C-level
- Lessons learned database

### Case Study 4: Program Manager xử lý dependencies

**Tình huống:**
Bạn là Program Manager cho 3 dự án song song:
- Dự án A: Migrate backend (6 tháng)
- Dự án B: Phát triển mobile app (4 tháng) — phụ thuộc vào Dự án A
- Dự án C: Training end-users (1 tháng) — phụ thuộc vào cả A và B

Dự án A bị trễ 3 tuần.

**Phân tích impact:**
```
A trễ 3 tuần
→ B bắt đầu muộn 3 tuần → B kết thúc muộn 3 tuần
→ C bắt đầu muộn 3 tuần → C kết thúc muộn 3 tuần
→ Go-live toàn bộ program trễ 3 tuần
```

**Hướng xử lý:**
1. Có thể B bắt đầu song song với phần không phụ thuộc A không?
2. Có thể crash schedule (thêm resource) cho A để recover không?
3. Communicate ngay với sponsor về impact và options

## Chứng Chỉ & Bằng Cấp {#chứng-chỉ}

### Lộ trình chứng chỉ theo cấp độ

```
Entry Level          Mid Level             Advanced
[CAPM]  [PSM I]  →  [PMP]  [PMI-ACP]  →  [PgMP]  [PfMP]
[Scrum  [Kanban]    [PRINCE2]              [PMO-CP]
 Fund.]              [PMI-RMP]
```

### Chi tiết từng chứng chỉ

#### CAPM (Certified Associate in Project Management)
- **Cấp độ:** Entry level
- **Tổ chức:** PMI (Project Management Institute)
- **Yêu cầu:** 23 giờ PM education, không cần kinh nghiệm
- **Đề thi:** 150 câu, 3 giờ
- **Chi phí:** ~$225 (member PMI) / $300 (non-member)
- **Phù hợp:** Mới bắt đầu, chưa có kinh nghiệm PM chính thức
- **Thời hạn:** 5 năm, renew bằng 15 PDU/năm

#### PMP (Project Management Professional)
- **Cấp độ:** Mid-level (tiêu chuẩn vàng ngành)
- **Tổ chức:** PMI
- **Yêu cầu:** 36 tháng kinh nghiệm PM (hoặc 60 tháng nếu không có bằng đại học) + 35 giờ PM education
- **Đề thi:** 180 câu, 230 phút (mix Predictive + Agile + Hybrid)
- **Chi phí:** ~$405 (member) / $555 (non-member)
- **Giá trị:** Tăng lương trung bình 16-25% theo PMI Salary Survey
- **Thời hạn:** 3 năm, renew 60 PDU

#### PSM I / CSM (Scrum Master)
- **PSM I** (Scrum.org): Không cần training, thi online, $150, không hết hạn
- **CSM** (Scrum Alliance): Cần training 2 ngày (~$500-1000), thi online, renew 2 năm
- **Phù hợp:** Developer muốn bước sang PM qua con đường Agile

#### PMI-ACP (Agile Certified Practitioner)
- **Yêu cầu:** 2000 giờ project experience + 1500 giờ Agile + 21 giờ Agile training
- **Chi phí:** ~$435 (member) / $495 (non-member)
- **Phù hợp:** PM làm môi trường Agile/Hybrid

#### PRINCE2 Foundation & Practitioner
- **Phổ biến:** UK, EU, Úc, một số công ty Nhật
- **Foundation:** Không yêu cầu kinh nghiệm, ~£280
- **Practitioner:** Cần Foundation, ~£350
- **Phù hợp:** Làm việc với công ty nước ngoài (Nhật, UK)

## Công Cụ & Phần Mềm {#công-cụ}

### Nhóm 1: Project Planning & Tracking

| Công cụ | Dùng khi nào | Free? | Ghi chú |
|---|---|---|---|
| **Jira** | Agile software projects | Có (≤10 users) | Industry standard cho dev team |
| **MS Project** | Traditional Waterfall, Gantt chart | Không | Phổ biến trong enterprise |
| **Asana** | Task management, team work | Có (basic) | Dễ dùng, phù hợp non-tech team |
| **Monday.com** | Flexible, visual boards | Có (basic) | Thân thiện với business team |
| **Notion** | Documentation + light PM | Có | Tốt cho early-stage startup |

## Case Studies Thực Tế {#case-studies}

### Case Study 5: NASA Mars Climate Orbiter (1999) — Failure

**Bối cảnh:**
Tàu vũ trụ trị giá $327 triệu bị mất vì lỗi đơn vị đo lường: một nhóm dùng metric (Newton), nhóm kia dùng imperial (pound-force).

**PM lessons:**
1. **Assumption is dangerous** — không bao giờ assume team hiểu giống nhau
2. **Interface agreements** phải được document rõ ràng
3. **Independent verification** quan trọng với các quyết định có rủi ro cao
4. **Communication gap** giữa các team là rủi ro dự án thực sự

**Áp dụng:** Trước khi integrate 2 hệ thống (ví dụ: API giữa frontend và backend), PM phải đảm bảo có Interface Agreement Document được cả 2 bên sign off.

---

### Case Study 6: Agile transformation tại Spotify

**Bối cảnh:**
Spotify không dùng Scrum thuần túy mà tạo ra model riêng với Squad, Tribe, Chapter, Guild.

**Cấu trúc:**
```
Squad  = nhóm nhỏ (~8 người), autonomous, 1 product area
Tribe  = tập hợp các squads cùng mission (~100 người)
Chapter = nhóm cùng skillset xuyên squad (designers, engineers)
Guild   = community of interest tự nguyện
```

**PM lessons:**
1. Không có phương pháp nào phù hợp tất cả — adapt và customize
2. **Autonomy** với team tăng engagement và tốc độ
3. Cân bằng giữa alignment (tribe) và autonomy (squad)
4. Văn hóa quan trọng hơn quy trình

---

### Case Study 7: Vietnam context — Dự án ERP cho công ty SME

**Bối cảnh:**
Công ty sản xuất 200 người ở Hà Nội, triển khai ERP lần đầu. PM là bạn.

**Thách thức đặc thù Việt Nam:**
1. **User resistance cao** — nhân viên quen Excel/giấy tờ, sợ mất việc
2. **Sponsor không hiểu IT** — giám đốc muốn "xong nhanh, rẻ"
3. **Data quality kém** — dữ liệu cũ không sạch, khó migrate
4. **Scope creep từ nhiều phòng ban** — mỗi phòng muốn customize riêng
5. **Key user hay vắng mặt** — khó lấy requirement

**Cách xử lý:**
| Thách thức | Chiến lược |
|---|---|
| User resistance | Change management: truyền thông lợi ích, training sớm, tìm "champion" trong từng phòng |
| Sponsor kỳ vọng phi thực tế | Education session đầu dự án: "đây là journey, không phải mua TV về cắm là xong" |
| Data quality | Dành 1-2 tháng data cleansing trước khi migrate, không để phút cuối |
| Scope creep | CAB (Change Advisory Board) họp 2 tuần/lần, mọi thay đổi phải qua đây |
| Key user vắng | Ký commitment letter, định nghĩa rõ "Project role" của họ ngay từ đầu |

## Bài Tập Thực Hành {#bài-tập}

### Bài tập 1: Viết Project Charter thực tế
**Yêu cầu:** Viết Project Charter cho một trong hai scenario:
- A) Dự án xây dựng blog cá nhân của bạn (technical PM perspective)
- B) Dự án migration một tính năng nhỏ từ hệ thống cũ sang mới tại công ty

**Phải bao gồm:** Objective, Scope IN/OUT, Milestones, Budget estimate, Risk top 3, Stakeholders

---

### Bài tập 2: Risk Register cho dự án thực tế
**Yêu cầu:** Lấy một dự án bạn đang tham gia (hoặc đã tham gia), viết Risk Register với ít nhất 8 rủi ro, đầy đủ: Probability, Impact, Score, Strategy, Action, Owner.

---

### Bài tập 3: EVM Calculation
**Scenario:**
- Budget: 500 triệu VND
- Duration: 20 tuần
- Tuần 10 review:
  - Kế hoạch xong 50% công việc
  - Thực tế xong 42% công việc
  - Đã chi 280 triệu

**Yêu cầu tính:**
1. PV, EV, AC
2. CV, SV
3. CPI, SPI
4. EAC (dự báo chi phí cuối dự án)
5. Nhận xét: dự án đang ở trạng thái nào?

*(Đáp án: PV=250M, EV=210M, AC=280M, CV=-70M, SV=-40M, CPI=0.75, SPI=0.84, EAC=667M)*

---

### Bài tập 4: Stakeholder Analysis
**Yêu cầu:**
1. Chọn một dự án (thực tế hoặc giả định)
2. Liệt kê tất cả stakeholders
3. Đánh giá trên ma trận Power/Interest:
   ```
   High Power │ Manage closely │ Keep satisfied │
              ├────────────────┼────────────────┤
   Low Power  │ Monitor        │ Keep informed  │
              └────────────────┴────────────────┘
                  Low Interest      High Interest
   ```
4. Xác định communication strategy cho từng nhóm

---

### Bài tập 5: Simulation — 1 tuần làm PM
**Yêu cầu:** Trong 1 tuần, thực hành các ritual của PM:
- Ngày 1: Viết Weekly Status Report cho dự án bạn đang tham gia (dưới góc nhìn PM)
- Ngày 2: Identify 3 risks trong tuần tới, assess và plan mitigation
- Ngày 3: Viết agenda cho 1 meeting sắp tới và facilitate nó
- Ngày 4: Làm retrospective 15 phút với team (Start/Stop/Continue)
- Ngày 5: Viết Lessons Learned cho một việc không suôn sẻ trong tuần

## Template & Checklist {#template}

### Project Kickoff Checklist
```
PRE-KICKOFF
☐ Project Charter được approve bởi sponsor
☐ Stakeholder list hoàn chỉnh
☐ Communication plan có trong project plan
☐ Team được assign và confirm availability
☐ Tools và workspace setup (Jira, Slack, Drive...)
☐ Project folder structure tạo sẵn

KICKOFF MEETING AGENDA (60-90 phút)
☐ Giới thiệu team (10 min)
☐ Trình bày Project Charter (15 min)
☐ Review timeline và milestones (15 min)
☐ Clarify roles và responsibilities (RACI) (15 min)
☐ Risk và assumptions (10 min)
☐ Communication norms (5 min)
☐ Q&A (10 min)
☐ Next steps và action items (5 min)

POST-KICKOFF
☐ Meeting minutes gửi trong 24h
☐ Action items được assign deadline
☐ Project plan update với thông tin từ meeting
```

### RACI Matrix Template
```
R = Responsible (người thực hiện)
A = Accountable (người chịu trách nhiệm cuối)
C = Consulted   (người được hỏi ý kiến)
I = Informed    (người được thông báo kết quả)

                 PM    Dev Lead   QA    Sponsor  Client
Kick-off         A/R     C        I       A        I
Requirement      C       R        C       A        R
Development      I       A/R      C       I        I
Testing          A       C        R       I        I
Go/No-go         C       C        C       A        R
Go-live          A/R     R        R       I        I
```

## Lời Kết

Chuyển từ Software Developer sang Project Manager là bước đi tự nhiên và có nhiều lợi thế. Background kỹ thuật của bạn sẽ giúp bạn:

1. **Giao tiếp tốt hơn** với dev team — bạn hiểu họ đang làm gì
2. **Risk assessment chính xác hơn** — bạn biết technical risk thực sự là gì
3. **Ước lượng thực tế hơn** — bạn không bị "ảo" về timeline
4. **Credibility cao hơn** — team tin tưởng PM hiểu kỹ thuật

Điều quan trọng nhất: **bắt đầu làm ngay**, đừng chờ "đủ kiến thức". PM là nghề học qua thực hành nhiều hơn lý thuyết.
